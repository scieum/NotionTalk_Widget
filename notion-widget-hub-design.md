# Notion 위젯 허브 — 통합 설계서

> **문서 성격**: Claude Code가 구현 시 참조하는 계획서. 상세 코드/파일 내용은 구현 단계에서 작성한다.
> **프로젝트 가칭**: `notion-widget-hub` (사이트명 미정 — 부록 D)
> **관련 문서**: 프론트 UI 표준 = `DESIGN.md`(Notion 디자인 시스템). 기존 `notion-widget-builder-design.md`는 **참고용으로만** 사용하며 본 설계서가 이를 대체한다.

---

## 1. 작업 컨텍스트

### 1.1 배경

기존 Notion 위젯 빌더 설계(시계·캘린더·유튜브·투두)를 참고하되 새로 재설계한다. 이번에는 **개인용 Notion 임베드 위젯**과 **교사용 수업 도구**라는 두 사용자 트랙을 하나의 사이트로 통합하고, 공부·운동 대시보드는 사용자의 Notion DB와 양방향 연동한다.

### 1.2 목적

- Notion 페이지에 임베드해 쓰는 위젯(뽀모도로·캘린더·대시보드·시계)을 URL 하나로 제공
- 교사가 수업 중 바로 쓰는 도구(수업 타이머·자리뽑기·랜덤뽑기)를 임베드/전체화면 양쪽으로 제공
- 대시보드 위젯은 사용자의 Notion 기록 DB를 읽어 시각화하고, 위젯에서의 기록 입력과 뽀모도로 완료 자동기록을 Notion에 쓴다

### 1.3 성격

**런타임 LLM이 없는 결정론적 웹앱.** "에이전트"는 빌더로서의 Claude Code만 가리킨다. 런타임 판단·검증은 전부 코드(zod 스키마, 순수함수, 사람 미리보기)로 처리한다.

### 1.4 범위

| 포함 | 제외 |
|------|------|
| 위젯 7종: 뽀모도로, 캘린더, 대시보드(공부/운동), 시계(아날로그/디지털), 수업 타이머, 자리뽑기, 랜덤뽑기 | 유튜브 플레이리스트·투두 위젯 (기존 설계에서 승계하지 않음, 추후 확장 후보) |
| 위젯 설정 빌더(설정 UI → 임베드 URL 생성) | 런타임 LLM, 위젯 마켓플레이스, 유료 결제 |
| Notion OAuth 공개 연동 + 대시보드 읽기/쓰기 | Notion `현재가` 주입류의 주기 배치 동기화 (이 프로젝트는 온디맨드만) |
| 뽀모도로 완료 → Notion 기록 DB 자동 append | 사용자 Notion 기존 행의 수정/삭제 |
| 임베드(iframe) + 독립 전체화면 페이지 양쪽 지원 | 네이티브 앱, 오프라인 PWA(1차 범위 밖) |

### 1.5 입출력 정의

- **입력**: 위젯 설정(URL 파라미터), 학생 명단(URL/localStorage), Notion OAuth 인가, Notion 기록 DB의 행들, 위젯에서의 기록 입력
- **출력**: 렌더링된 위젯 화면, 임베드 URL, Notion 기록 DB에 append되는 행(기록 입력·뽀모도로 완료), 대시보드 차트/통계

### 1.6 주요 제약조건

1. **Notion API는 브라우저 직접 호출 불가**(CORS + 토큰 노출) → 서버 프록시 필수
2. **Notion 임베드 = iframe** → 서드파티 쿠키 차단(Safari 기본, Chrome 진행 중)으로 쿠키 세션 인증이 임베드 안에서 동작하지 않음 → **서명된 위젯 토큰** 방식 필요 (§3.3)
3. **Notion 레이트리밋 ~3 req/s** — 온디맨드 요청도 서버 캐시로 완충
4. 공개 배포이므로 사용자별 Notion DB 스키마가 제각각 → **공식 템플릿 + 속성 매핑** 규약 필요 (§5.3)
5. 학생 명단은 개인정보 — URL 인코딩 시 노출 리스크 관리 (§4.7)
6. Notion 무연동·비로그인 상태에서도 대시보드를 제외한 모든 위젯은 완전 동작해야 함(정적 우선 원칙)

### 1.7 용어

| 용어 | 정의 |
|------|------|
| 위젯 | Notion 임베드 또는 전체화면으로 렌더링되는 단일 기능 화면 |
| 빌더 | 위젯을 고르고 설정한 뒤 임베드 URL을 발급받는 사이트 홈 |
| 위젯 토큰 | 대시보드 임베드 URL에 포함되는 서명 토큰. 서버가 이것으로 사용자·연동 DB를 식별 |
| 기록 DB | 사용자 Notion 워크스페이스의 공부/운동 기록 데이터베이스(공식 템플릿 복제 권장) |
| 설정 문자열 | 위젯 설정 JSON을 압축·인코딩해 URL 쿼리에 넣은 값 |

## 3. 아키텍처

### 3.1 구성도 — 두 계층 (분리가 핵심)

```
[정적 계층 — 서버 불필요]
  브라우저(React 위젯) ← 설정은 URL/localStorage에서만 읽음
  뽀모도로 · 캘린더 · 시계 · 수업타이머 · 자리뽑기 · 랜덤뽑기
                │
                │ (대시보드/자동기록만 통과)
                ▼
[연동 계층 — Node+Express 백엔드]
  ├ OAuth 라우트(Notion 공개 통합: 인가 → 토큰 교환 → 저장)
  ├ 위젯 토큰 발급/검증(서명, iframe 인증용)
  ├ Notion 프록시 API(대시보드 읽기 query · 기록 append)
  ├ 단기 캐시(읽기 TTL 30~60초 — 레이트리밋 완충)
  └ 스토어(SQLite: 사용자·암호화 토큰·DB 매핑·위젯 토큰)
                │
                ▼
        [Notion API (~3 req/s)]
```

- **정적 계층이 기본**: 위젯 7종 중 6종은 백엔드 없이 완결. 서버 장애 시에도 대시보드 외 전부 정상 동작해야 한다.
- **연동 계층은 온디맨드만**: 주식 위젯과 달리 cron 주기 동기화가 없다. 대시보드가 열릴 때 읽고, 기록이 발생할 때 쓴다.

### 3.2 기술 스택

- **모노레포**: npm workspaces (`notion-widget-hub`)
- **웹**: TypeScript + React + Vite, DESIGN.md 디자인 토큰
- **서버**: TypeScript + Node + Express, better-sqlite3, zod
- **테스트**: Vitest (core 순수함수 필수)
- **호스팅**: 웹 = 정적 배포(Vercel 또는 Railway static), 서버 = Railway 단일 서비스 (주식 위젯과 동일 패턴 — 운영 경험 재사용)

### 3.3 인증 모델 (핵심 설계 결정)

| 컨텍스트 | 인증 방식 | 근거 |
|----------|-----------|------|
| 빌더(사이트 직접 방문) | Notion OAuth → 세션 쿠키(퍼스트파티) | 표준 로그인 흐름 |
| 임베드 위젯(iframe) | **위젯 토큰**(URL에 포함, HMAC 서명) | 서드파티 쿠키가 iframe에서 차단되므로 쿠키 불가. Indify 등 기존 Notion 위젯 서비스와 동일 패턴 |

- 흐름: 빌더에서 로그인 → 기록 DB 연결 → "대시보드 위젯 URL 발급" → URL 안의 위젯 토큰으로 서버가 사용자를 식별
- 위젯 토큰은 **해당 위젯·해당 사용자·읽기/쓰기 범위로 한정**. 재발급 시 이전 토큰 무효화(유출 대응).
- Notion access token은 서버 SQLite에 **암호화 저장**(키는 환경변수). 브라우저에는 절대 내려보내지 않는다.

### 3.4 라우팅

```
/                      빌더 홈(위젯 갤러리 + 설정 UI + URL 발급)
/w/:widget?c=...       임베드용 (컴팩트 레이아웃)
/f/:widget?c=...       전체화면용 (프로젝터 레이아웃 — 대형 타이포)
```

- 동일 위젯 컴포넌트가 `embed | fullscreen` 레이아웃 모드만 달리 렌더링. 설정 문자열 `c`는 공유.
- 임베드 페이지에서 "전체화면으로 열기" 버튼 제공(같은 `c`로 `/f/` 이동) — "둘 다 지원"의 구현.

## 4. 위젯 사양

공통: 모든 위젯은 DESIGN.md 토큰 적용, 다크모드 지원, 이모지 대신 SVG 아이콘/컬러 닷.

### 4.1 시계
- 아날로그/디지털 모드, 12/24시간제, 날짜 표시 토글, 크기·색 변형
- 저장: 설정 문자열만. 서버 불필요.

### 4.2 캘린더
- **순수 표시형 월간 달력**(오늘 강조, 주 시작 요일 설정). Notion 캘린더 연동은 1차 범위 밖 — 근거: 연동 시 대시보드와 동일한 OAuth·매핑 비용이 추가되는데 표시형만으로 임베드 수요 대부분을 충족. 확장 후보로만 기록(부록 D).

### 4.3 뽀모도로
- 집중/휴식 길이, 자동 반복, 알림음, 사이클 카운트 설정
- **세션 기록**: 비연동 시 localStorage에 일별 누적. Notion 연동 + 자동기록 ON이면 세션 완료 시 기록 DB에 append (§5.4)
- 타이머 로직은 `packages/core` 상태머신(순수함수) — 탭 비활성 시 드리프트 보정(타임스탬프 기준 계산, setInterval 누적 금지)

### 4.4 대시보드 (공부/운동)
- Notion 기록 DB를 읽어 기간별(일/주/월) 누적 시간·횟수·목표 대비 진행률을 차트로 표시
- 위젯 내 **빠른 기록 입력**(예: "공부 50분, 수학") → Notion append
- 뽀모도로 자동기록 행도 동일 DB에 쌓여 함께 집계됨
- 집계 계산은 core 순수함수(서버·클라이언트 공용) — Notion에는 원시 행만, 계산은 코드가

### 4.5 수업 타이머
- 카운트다운(프리셋: 5/10/15분 등 + 직접 입력), 종료 알림음·화면 플래시, 일시정지
- 전체화면 모드에서 교실 뒤에서도 보이는 초대형 숫자(뷰포트 기준 vw 타이포)

### 4.6 자리뽑기
- 교실 격자(행×열, 기본 5×6) 정의 + 사용 안 하는 칸 비활성화
- 학생 명단 → 랜덤 배정. **고정석·분리 조건(특정 학생 앞자리 고정, 특정 쌍 이웃 금지)** 지원
- 뽑기 애니메이션(순차 공개) — 수업 연출용
- 배정 알고리즘은 core 순수함수: 시드 기반 셔플 + 제약 충족(제약 불능 시 명시적 실패 반환, 무한루프 금지)
- 결과는 localStorage에 최근 N회 저장(이전 배치 다시 보기)

### 4.7 랜덤뽑기 + 명단 저장 규약
- 명단에서 N명 추첨, 중복 허용/제외 토글, 이미 뽑힌 학생 표시, 애니메이션
- **명단 저장 규약(자리뽑기 공통)**: 확정 사항 6은 "URL 기본"이나, 학생 **실명이 URL에 노출**되면 링크 공유·브라우저 히스토리를 통해 유출될 수 있다. 따라서 명단만 예외 처리:
  - 기본: **localStorage** (명단 프리셋 이름으로 여러 반 저장)
  - 선택: URL 인코딩(기기 간 이동용) — 발급 시 "학생 이름이 링크에 포함됩니다" 고지 + 출석번호/이니셜 사용 권장 문구
  - 명단 외 일반 설정(격자 크기, 타이머 길이 등)은 원칙대로 URL 기본

### 4.8 설정 문자열 규약 (전 위젯 공통)
- 위젯별 설정 스키마는 zod로 정의(`packages/core`) → JSON → lz-string 압축 → base64url → `?c=`
- 디코딩 실패·스키마 불일치 시 **기본 설정으로 폴백 + 화면 내 안내 배지** (빈 화면 금지)
- localStorage는 보조: 빌더에서 마지막 설정 자동 복원, 명단 프리셋 저장

## 5. 워크플로우 정의

런타임 LLM이 없으므로 "LLM 판단 영역"은 전무하다. 아래 표의 판단 주체는 전부 코드이며, LLM(Claude Code)의 판단은 **빌드 타임**(구현 순서·리팩터링)에만 존재한다.

### 5.1 W1 — 위젯 발급·렌더링 (정적 계층)

```
빌더에서 위젯 선택 → 설정 UI → [미리보기] → URL 발급(복사)
→ Notion에 /embed 붙여넣기 → 위젯 로드 → c= 디코딩 → 렌더
                                  └ 실패 → 기본 설정 폴백 + 배지
```

| 단계 | 성공 기준 | 검증 방법 | 실패 시 |
|------|-----------|-----------|---------|
| 설정 인코딩 | 인코딩→디코딩 왕복 동일성 | 스키마 검증(zod) + core 왕복 단위테스트 | 발급 차단 + 입력 오류 표시 |
| 위젯 로드 | 1초 내 첫 렌더(정적 위젯) | 규칙 기반(Lighthouse/수동) | — |
| 디코딩 | zod 파싱 성공 | 스키마 검증 | **스킵+폴백**: 기본 설정 렌더 + 배지 |

### 5.2 W2 — Notion 연동 온보딩

```
빌더 로그인(OAuth) → 워크스페이스 인가 → 기록 DB 선택
→ 속성 매핑(자동 → 실패 시 수동 UI) → 매핑 저장 → 위젯 토큰 발급
```

| 단계 | 성공 기준 | 검증 방법 | 실패 시 |
|------|-----------|-----------|---------|
| 토큰 교환 | access token 수신·암호화 저장 | 스키마 검증(응답 zod) | 자동 재시도 1회 → 에러 화면(재인가 유도) |
| DB 선택 | DB 접근 권한 확인(query 1회 성공) | 규칙 기반 | **에스컬레이션**: "통합에 페이지 공유했는지" 안내 |
| 속성 매핑 | 필수 속성 4종 매핑 완료(§5.3) | 스키마 검증 | 자동 매핑 실패 → **수동 매핑 UI**(사람 검토) |
| 토큰 발급 | 서명 검증 왕복 성공 | 규칙 기반 + 단위테스트 | 발급 차단 |

### 5.3 속성 매핑 규약 (공개 배포의 핵심 문제)

사용자마다 DB 스키마가 다르므로:

- **공식 Notion 템플릿 제공**(복제 권장 경로) — 필수 속성: `날짜`(date) · `분류`(select: 공부/운동/뽀모도로 등) · `시간(분)`(number) · `메모`(rich_text, 선택)
- 자동 매핑: 속성 **타입 + 이름 후보군**(한/영)으로 추정 → 미리보기로 사용자 확인
- 매핑 결과는 서버 SQLite에 사용자·DB 단위로 저장. 이후 모든 읽기/쓰기는 매핑 테이블을 통해서만 속성 접근(하드코딩 금지)

### 5.4 W3 — 대시보드 읽기 / 기록 쓰기 (연동 계층)

```
[읽기] 위젯 로드 → 위젯 토큰 검증 → 캐시 확인(TTL 30~60초)
      → miss 시 Notion query(기간 필터) → zod 검증 → core 집계 → 차트
[쓰기] 기록 입력 or 뽀모도로 완료 → 서버 append API → Notion 행 생성
      → 성공 응답 → 위젯 낙관적 갱신 확정
```

| 단계 | 성공 기준 | 검증 방법 | 실패 시 |
|------|-----------|-----------|---------|
| 토큰 검증 | 서명·범위·만료 통과 | 규칙 기반 | 401 + 위젯에 "재발급 필요" 안내 |
| Notion 읽기 | 200 + 행 스키마 통과 | 스키마 검증 | 재시도 3회(지수 백오프) → **직전 캐시 표시 + 오프라인 배지** |
| 집계 | NaN 없음, 합계 = Σ행 | core 단위테스트 | 빌드 차단(테스트) |
| Notion 쓰기 | 201 + 필수 속성 채움 | 스키마 검증 + 응답 확인 | 재시도 3회 → **localStorage 대기열 보관 + 재전송 버튼**(기록 유실 금지) |
| 뽀모도로 자동기록 | 세션당 정확히 1행 | 중복키 = 세션ID(클라이언트 UUID) | 중복 감지 시 스킵 + 로그 |

### 5.5 상태 전이 (연동 위젯)

`ANONYMOUS → CONNECTED → (DEGRADED ⇄ CONNECTED) → REVOKED`

- `DEGRADED`: Notion 호출 연속 실패 — 캐시 표시 유지, 쓰기는 대기열로
- `REVOKED`: 사용자가 Notion에서 통합 해제 — 재인가 유도, 로컬 기능은 계속 동작

## 6. 도메인 불변식 (위반 금지)

1. **append-only**: 시스템은 기록 DB에 **행 생성만** 한다. 사용자 기존 행의 수정·삭제 절대 금지. 시스템 생성 행도 생성 후에는 건드리지 않는다(수정은 사용자 몫).
2. **필드 소유권**: 시스템이 채우는 속성은 매핑된 필수 속성(+ 선택 `출처` 태그)뿐. 그 외 속성은 쓰지 않는다.
3. **집계는 코드, 원본은 Notion**: 통계·진행률은 core 순수함수가 계산해 화면에만 표시. 집계값을 Notion에 역기록하지 않는다.
4. **중복 방지**: 뽀모도로 자동기록 중복키 = 클라이언트 세션 UUID. 동일 UUID 재전송은 멱등 처리.
5. **정적 우선**: 대시보드 외 위젯은 어떤 코드 경로에서도 서버에 의존하지 않는다(빌드 시 import 경계로 강제).
6. **시크릿 경계**: Notion 토큰·서명 키는 서버에만. 위젯 토큰은 URL에 있으므로 **범위 최소화 + 재발급 무효화**가 유일한 방어선임을 전제로 설계.
7. **명단 개인정보**: 학생 명단의 서버 전송·저장 금지(전 기능 클라이언트 완결). URL 인코딩은 명시적 고지 후에만.

## 8. 정책 수치 (초기 기본값)

- 대시보드 읽기 캐시 TTL **30~60초** · Notion 재시도 **3회**(지수 백오프) · Notion **~3 req/s** 준수
- 위젯 토큰 만료 **무기한(재발급 시 무효화)** — 임베드 URL은 갱신이 어려우므로
- 자리뽑기 제약 충족 시도 상한 **1,000회**(초과 시 "조건 완화 필요" 반환)
- 명단 프리셋 localStorage 최대 **10개** · 자리배치 히스토리 **5회**

## 9. 빌드 순서 (버티컬 슬라이스)

1. **core + 시계**: 설정 코덱(왕복 테스트) → 시계 위젯 → `/w/`·`/f/` 라우팅 → 정적 배포 (파이프라인 최소 완성)
2. **수업 도구**: 자리배정·추첨 core + 단위테스트 → 자리뽑기·랜덤뽑기·수업 타이머 UI → 명단 저장 규약
3. **뽀모도로**: 상태머신 core + 테스트 → UI + localStorage 기록
4. **서버 기반**: OAuth → 토큰 암호화 저장 → 위젯 토큰 발급/검증 → Notion 프록시 골격
5. **대시보드**: 템플릿 DB + 속성 매핑 → 읽기·집계·차트 → 기록 쓰기 + 실패 대기열 → 뽀모도로 자동기록 연결
6. **빌더 홈**: 위젯 갤러리 + 설정 UI + URL 발급 + 온보딩 흐름
7. **종단 테스트**: 실제 Notion 워크스페이스에서 온보딩→임베드→기록→집계 전 과정
